# Guidance Rules

用于判断一句要求是否应该写进 `AGENTS.md` 或 `CLAUDE.md`，以及如何写成短的 agent 项目约束。

## 写入判断

写入：

- 作者明确说“写进 AGENTS/CLAUDE”“以后 agent 都要/不要这样做”。
- 规则约束 agent 行为：读写顺序、确认条件、验证要求、工具偏好、输出方式。
- 文件边界：禁止触碰、不要新建、只能在特定条件下修改。
- 项目级协作约定：问法、汇报方式、提交要求、技能路由偏好。

不写入：

- 小说设定、人物档案、世界观、时间线、梗概、调研摘录。
- 章节进度、修订过程、迁移记录、检查点；转 `$ns-canon` 写入 `novel-studio/records/`。
- 风格样例、长说明或具体剧情事实；转 `$ns-canon` 写 notes/YAML。
- 单次反馈或临时试写方向，除非作者明确要固化成 agent 规则。

## 作者协作

- 作者没有明确拍板时，先用问题或候选写法确认。
- 可以帮作者压缩表达，但不要把猜测写成硬规则。
- 对“不要随便”“尽量”等模糊词，保留弹性：写成优先级或触发条件。

## 高效写法

- 用动词开头：保持、避免、优先、不要、允许、待确认。
- 一条只写一个动作或边界。
- 删除长原因，保留可执行结果。
- 合并同类项，去掉重复词。
- 软偏好写成“优先/默认/除非作者要求”，硬禁区写成“不要/禁止”。

## 推荐小节

已有小节优先沿用。没有时可使用：

- `## 必须遵守的工作流`
- `## 不要写/不要改`
- `## 协作偏好`
- `## 工具和验证`
- `## 待确认`

如果目标文件已有项目规范结构，不强行改标题；可新增 `## Agent Project Constraints` 作为最小容器。

## 去向

- agent 项目约束写入 `AGENTS.md` 或 `CLAUDE.md`。
- 小说事实、计划、进度、来源和长说明转 `$ns-canon`，不要在本 skill 里另判一套位置。
- AGENTS/CLAUDE 只保留能影响 agent 后续执行方式的原则；具体故事事实在记忆文件里做唯一来源。
