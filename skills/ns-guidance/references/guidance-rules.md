# Guidance Rules

用于判断一句要求是否应该写进 `AGENTS.md` 或 `CLAUDE.md`，以及如何写得短而有用。

## 写入判断

写入：

- 作者明确说“以后都要”“不要再写”“不能改”“保持这种味道”。
- 规则会影响多章、多卷或整本书。
- 内容短小，不值得单独创建 notes。
- 能约束后续生成：题材禁区、口吻偏好、结构偏好、尺度边界。
- 作者仍在确认但需要持续提醒的事项，先写进“待确认”。

不写入：

- 单章修订意见、一次性反馈、临时试写方向。
- 人物长档、世界观长说明、时间线、梗概、调研摘录。
- 尚未确认且会强烈改变故事方向的设定；先交给 `$ns-ideate` 或 `$ns-build` 讨论。

## 作者协作

- 作者没有明确拍板时，先用问题或候选写法确认。
- 可以帮作者压缩表达，但不要把猜测写成硬规则。
- 对“不要随便”“尽量”等模糊词，保留弹性：写成优先级或触发条件。

## 高效写法

- 用动词开头：保持、避免、优先、不要、允许、待确认。
- 一条只写一个动作或边界。
- 删除长原因，保留可执行结果。
- 合并同类项，去掉重复词。
- 软偏好写成“优先/默认/除非作者要求”，硬禁区写成“不要/禁止”。

## 推荐小节

已有小节优先沿用。没有时可使用：

- `## 必须遵守`
- `## 不要写/不要改`
- `## 风格偏好`
- `## 内容边界`
- `## 结构偏好`
- `## 待确认`

如果目标文件已有项目规范结构，不强行改标题；可新增 `## Novel Studio Guidance` 作为最小容器。

## 去向

- 短全局指导：写入 `AGENTS.md` 或 `CLAUDE.md`。
- 长人物、世界、时间线、梗概、样文：写入 `novel-studio/notes/*.md`。
- 结构化事实、计划、索引、来源边界：写入对应 YAML。
- 调研和修订过程：写入 `novel-studio/logs/*.md`。
